# Use IAM Identity Policies for S3 Write Access

## Status

Accepted

## Date

2026-07-16

---

# Context

The Aviation Operations Data Platform uploads raw OpenSky API responses into an Amazon S3 data lake.

The ingestion application authenticates using AWS credentials and uploads files using the Boto3 SDK.

The platform required permission to write objects into the following location:

raw/opensky/states/

---

# Problem

Where should S3 write permissions be granted?

Two approaches were considered.

---

# Option 1 — IAM Identity Policy ✅

Attach permissions directly to the IAM identity used by the application.

Example:

- IAM User (Local Development)
- ECS Task Role (Production)
- EMR Role
- Lambda Role

### Advantages

- Least privilege
- Easy auditing
- Works naturally with Boto3
- Recommended AWS practice
- Easy migration from local development to ECS

### Disadvantages

- Each application identity requires its own policy.

---

# Option 2 — Bucket Policy

Grant permissions from the S3 bucket itself.

### Advantages

- Useful for cross-account access
- Centralized resource permissions
- Supports public or restricted bucket scenarios

### Disadvantages

- Requires Principal declarations
- More complex for single-account applications
- Not required for this project

---

# Decision

The project will use IAM identity-based policies.

Local development uses an IAM user.

Future production deployments will use IAM roles attached to AWS compute services.

---

# Rationale

The application accesses S3 using Boto3.

Boto3 authenticates as an AWS identity.

That identity should possess only the permissions required to perform its task.

The application currently requires only:

- s3:PutObject

within the raw OpenSky prefix.

---

# Lessons Learned

An AccessDenied error occurred during development.

The initial IAM policy allowed uploads to:

raw/opensky/states/*

However, the application attempted to upload to:

raw/opensky/

without the states folder.

Updating the application prefix resolved the authorization failure.

This reinforced the importance of matching IAM resource ARNs with the actual S3 object keys generated by the application.

---

# Future Considerations

Future pipeline stages may additionally require:

- s3:GetObject
- s3:ListBucket
- s3:DeleteObject (optional)
- KMS permissions if server-side encryption is enabled.
